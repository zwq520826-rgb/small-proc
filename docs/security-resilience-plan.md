# 小程序云服务安全与韧性治理计划

## 1. 目标
- 防止周期性接口扫描导致服务抖动、成本异常和安全事件。
- 建立统一的限流、降级、日志与告警机制。
- 在不大改现有业务架构的前提下，优先保护登录、下单、支付、管理端接口。

## 2. 范围
- 云函数：`order-service`、`payment-service`、`wechat-pay-notify`、`wechat-refund-notify`、`admin-service`。
- 小程序端：关键页面降级提示（首页、下单、订单中心）。
- 管理端：安全配置面板（阈值、开关、状态查看）。

## 3. 实施阶段

### Phase 0（当天）基线与风险收口
- 下线/收敛敏感日志：手机号、身份证、支付密文、完整交易报文只保留脱敏字段。
- 给所有接口统一补 `requestId`，日志按 `requestId` 串联。
- 产出接口清单：公开接口、登录接口、支付回调接口、管理接口分级。

### Phase 1（1-3 天）统一限流
- 新增通用限流器（云函数公共模块）：
  - 维度：`IP + 接口`、`UID + 接口` 双维度。
  - 策略：滑动窗口（如 60s）。
  - 超限返回：`429` + 标准错误码 `RATE_LIMITED`。
- 差异化阈值：
  - 登录/管理登录：严格（例：5 次/5 分钟）。
  - 订单查询：中等（例：60 次/分钟）。
  - 支付回调：高阈值 + 幂等保护。

### Phase 2（3-5 天）降级与熔断
- 新增运行时开关（`maintenance_settings`）：
  - `degrade_home_read`
  - `degrade_dashboard_read`
  - `degrade_non_core_write`
- 首页、看板优先读缓存（`daily_stats` / 本地缓存），下游慢时返回降级数据。
- 对外部依赖（支付、实名）增加超时与失败回退路径，避免级联阻塞。

### Phase 3（5-7 天）日志、告警、演练
- 结构化日志字段：`time`、`level`、`service`、`api`、`uid`、`ip_hash`、`requestId`、`errCode`、`latencyMs`。
- 告警指标：
  - `429` 比例
  - `5xx` 比例
  - 回调验签失败率
  - DB 慢查询与云函数超时率
- 压测与演练：
  - 模拟接口扫描
  - 验证限流命中
  - 验证降级开关生效与恢复

## 4. 技术设计要点
- 幂等：支付/退款回调按 `out_trade_no` / `out_refund_no` 去重。
- 安全：管理端接口强制鉴权 + 严格限流，必要时加 allowlist。
- 数据：统计走增量计数，避免全表扫描。
- 合规：日志与导出默认脱敏，敏感信息仅最小化存储。

## 5. 验收标准
- 接口被扫描时，核心下单/支付成功率不低于 99%。
- 限流命中后服务无雪崩，云函数成本峰值可控。
- 所有关键接口具备 `requestId` 串联日志。
- 管理端可一键开启/关闭降级开关并可视化当前状态。

## 6. 回滚与应急
- 所有限流/降级策略支持配置化关闭（不发版）。
- 出现误伤时：先放宽阈值，再按 IP/UID 局部豁免。
- 支付链路异常优先保障回调可重试与业务幂等，不做破坏性处理。

5. 在支付回调函数增加标准化日志与告警埋点。
6. 编写压测脚本与发布前检查清单。
