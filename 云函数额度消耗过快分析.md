# 云函数额度消耗过快分析

## 一、现象与核心原因

云函数额度消耗过快，通常不是单点问题，而是由以下几类因素叠加导致：

1. `clientDB` 底层同样依赖云函数执行，每次查询/写入都会计入云函数调用次数。  
2. 云函数计费与实际执行时长（毫秒级）和内存配置相关，执行越慢、内存越大，`GBs` 消耗越高。  
3. 云函数拆分过细会放大“保底消耗”，函数数量越多，固定成本越高。  

---

## 二、关键计费认知（结合当前业务）

### 1）`clientDB` 不等于“零调用成本”

- 前端使用 `clientDB` 虽然开发便捷，但请求最终仍走云端逻辑。  
- 高频页面（列表轮询、反复进入页面、无缓存刷新）会把调用次数快速拉高。  
- 需要把 `clientDB` 请求当作“云函数调用”来管理，而不是普通本地读写。

### 2）执行时间按 ms 计费，优化时长直接省钱

- 云函数执行时间精确到毫秒，耗时每下降一部分，`GBs` 会同步下降。  
- 默认内存为 `512M`，若函数只做轻量查询/组装返回，内存可降至更低档位。  
- 在满足稳定性的前提下，内存降档 + 代码提速 = 双重降费。

### 3）函数数量影响保底费用，合并可显著降本

- 将多个业务函数合并为一个入口（例如云对象、路由插件）可减少保底消耗。  
- 示例：  
  - 10 个函数：极端情况日保底约 `10 * 0.24 = 2.4元`（约 `21600 GBs`）  
  - 1 个函数：极端情况日保底约 `1 * 0.24 = 0.24元`（约 `2160 GBs`）  
- 在逻辑允许范围内，函数合并越充分，固定消耗越低。

---

## 三、优化方案（按优先级执行）

## P0：先控“调用次数”

### 1）减少无效请求

- 页面进入时做“去重触发”，避免 `onShow`、`onLoad`、下拉刷新、定时器同时请求。  
- 对同参数请求做短时间防抖/节流（例如 300ms~1s）。  
- 避免列表页频繁“全量刷新”，改为增量更新（仅拉取变化数据）。

### 2）增加本地缓存

- 对读多写少的数据做本地缓存（例如 10~60 秒短缓存）。  
- 二次进入页面优先读缓存，再异步刷新云端。  
- 配合 `etag/version` 或 `updatedAt` 判断是否需要重新拉取。

### 3）合并前端多次查询

- 一次页面渲染需要多块数据时，优先走单个聚合接口返回（减少 N 次调用）。  
- 例如：用户信息 + 统计数据 + 列表摘要，改为一次返回组合结构。

## P1：再控“单次执行成本（GBs）”

### 1）优化云函数执行路径

- 把慢逻辑从主链路剥离（如日志分析、非实时统计改异步任务）。  
- 数据库查询只取需要字段，避免 `*` 全字段读取。  
- 为高频查询条件建立索引，减少扫描耗时。  
- 避免函数内重复查询同一数据，优先做函数内短缓存。

### 2）优化内存档位

- 基于函数类型分档：  
  - 轻量读写：优先低内存档；  
  - 复杂计算/大对象处理：保留高内存档。  
- 调整后对比：错误率、超时率、平均耗时，确保稳定后再固化配置。

### 3）控制返回体积

- 返回数据只保留前端真正使用字段。  
- 列表接口增加分页与上限（如 `pageSize` 上限），避免一次返回过大。  
- 大字段（富文本、长日志）改按需单独拉取。

## P2：重构函数组织方式

### 1）业务函数合并

- 用“云对象”或“路由插件”将多个小函数合并为统一入口。  
- 建议按业务域分组：订单域、骑手域、用户域，而不是每个动作一个函数。  
- 统一鉴权、参数校验、异常处理，减少重复代码与重复初始化开销。

### 2）冷热路径拆分

- 高频核心路径保持极简。  
- 低频且耗时任务（导出、报表、历史重算）放入独立异步函数，避免拖慢高频函数。

---

## 四、建议的落地节奏（1~2 周）

### 第 1 阶段：基线采集（1~2 天）

- 统计 TOP 调用函数：调用次数、平均耗时、P95 耗时、失败率。  
- 标记高频页面请求链路，找出重复调用点。  
- 识别可合并函数清单（按业务域）。

### 第 2 阶段：快速降本（3~5 天）

- 完成前端去重请求、短缓存、聚合接口改造。  
- 对 TOP 函数做查询字段裁剪、索引优化、逻辑裁剪。  
- 试点下调 1~2 个轻量函数内存档，观察稳定性。

### 第 3 阶段：结构优化（3~5 天）

- 引入云对象/路由插件完成函数合并。  
- 建立统一入口层（鉴权、限流、日志、错误码）。  
- 对低频重任务改异步化，减少主链路占用。

---

## 五、持续监控指标（必须长期跟踪）

- 每日云函数调用次数（总量 + 按函数分布）  
- 每日 `GBs` 消耗（总量 + 按函数分布）  
- 平均耗时、P95 耗时、超时率、失败率  
- 高频页面的人均请求次数（进入一次页面触发多少次云调用）  
- 函数合并前后保底费用对比

---

## 六、结论

要解决“云函数额度消耗过快”，必须同时做三件事：

1. 把 `clientDB` 请求按云函数调用来治理，先压调用次数；  
2. 优化执行耗时和内存配置，降低单次 `GBs`；  
3. 合并业务函数，减少函数数量带来的保底开销。  

按上述顺序推进，通常可以在不影响业务功能的前提下，实现明显的调用量和费用下降。

## 七、具体优化方案（可直接执行）

## 方案A：前端请求治理（目标：调用次数下降 30%~60%）

### A1. 页面只保留一个“主触发入口”

- 问题：`onLoad`、`onShow`、下拉刷新、手动刷新都在请求，导致重复调用。  
- 改造：统一走 `refreshPageData()`，其他入口只触发它，不直接请求。  
- 验收：
  - 同一页面一次进入，主接口调用次数 `<= 1`；
  - 连续返回页面 5 次，总请求次数下降 30% 以上。

示例（思路）：

```js
let loading = false
let lastReqAt = 0
const MIN_INTERVAL = 800

async function refreshPageData(force = false) {
  const now = Date.now()
  if (loading) return
  if (!force && now - lastReqAt < MIN_INTERVAL) return
  loading = true
  lastReqAt = now
  try {
    await fetchPageData()
  } finally {
    loading = false
  }
}
```

### A2. 给读多写少接口加短缓存

- 问题：每次进页面都打云端，数据变化不大却重复扣次数。  
- 改造：本地缓存 10~60 秒，先读缓存再后台刷新。  
- 验收：
  - 10 秒内重复进入页面，云函数调用次数减少 50% 以上；
  - 页面首屏时间不劣化。

### A3. 聚合接口替代多个接口

- 问题：一个页面首次渲染调用 3~6 个函数。  
- 改造：新增一个聚合函数，返回页面所需组合数据。  
- 验收：
  - 页面首屏云调用由 N 次降为 1 次；
  - 首屏总耗时下降 20% 以上。

## 方案B：云函数执行优化（目标：单次 GBs 下降 20%~40%）

### B1. 数据查询“只取必要字段”

- 问题：查询返回大量无用字段，增加传输和序列化时间。  
- 改造：所有高频查询明确字段白名单。  
- 验收：
  - 高频函数平均响应包体下降 30%；
  - 平均执行时长下降 15% 以上。

### B2. 建立并使用高频索引

- 问题：条件查询走全表扫描，P95 高。  
- 改造：按常见筛选条件建索引（如 `riderId + status + updateTime`）。  
- 验收：
  - P95 耗时下降 30%；
  - 慢查询占比明显下降。

### B3. 下调轻量函数内存

- 问题：轻量查询也使用默认 `512M`。  
- 改造：把纯查询、简单写入函数逐步降到更低内存档（分批灰度）。  
- 验收：
  - 错误率和超时率不升高；
  - 同调用量下每日 `GBs` 明显下降。

实施建议：

1. 先选 2 个低风险函数试点；  
2. 观察 2~3 天（错误率、超时率、平均耗时）；  
3. 再批量迁移其他轻量函数。

## 方案C：函数结构重构（目标：保底费用降低）

### C1. 按业务域合并函数

- 问题：函数拆分过细，数量大，保底成本高。  
- 改造：以“业务域”为单位合并（例如 `rider`、`order`、`user`）。  
- 验收：
  - 函数总数下降（例如 10 -> 3）；
  - 日保底费用按函数数等比例下降。

推荐路由结构：

```js
// 云对象/路由入口示意
{
  "rider.getHallData": handlerA,
  "rider.getTaskList": handlerB,
  "rider.acceptTask": handlerC
}
```

### C2. 统一公共能力，减少重复开销

- 在统一入口做：鉴权、参数校验、错误码、日志采集。  
- 每个子处理函数只做业务逻辑，避免重复初始化。  
- 验收：
  - 重复代码行数下降；
  - 函数初始化时间和失败率下降。

## 方案D：针对当前业务页面的改造清单（可本周执行）

### D1. 骑手大厅页（`hall`）

- 合并“统计信息 + 当前状态 + 待处理数量”为一个聚合接口。  
- 页面 30 秒内重复进入优先读缓存。  
- 下拉刷新才强制拉最新数据（`force = true`）。

### D2. 任务列表页（`tasks/list`）

- 分页接口强制 `pageSize` 上限（如 20/30）。  
- 增量拉取（基于 `updatedAt` 或最后一条游标），避免每次全量。  
- 筛选条件变化时才重新请求，不变更条件不重复调用。

### D3. 公共请求层

- 增加请求去重 key：`接口名 + 参数摘要`。  
- 1 秒内同 key 只放行一次。  
- 请求失败后的重试改为指数退避，避免瞬时风暴。

## 九、量化目标（避免“做了但看不出效果”）

- 7 天内：调用次数下降 `>= 30%`  
- 7 天内：平均执行时长下降 `>= 20%`  
- 14 天内：每日 `GBs` 消耗下降 `>= 35%`  
- 14 天内：函数数量下降 `>= 40%`（按业务实际可调整）

未达到目标时，优先复查三项：  
1）页面是否仍有重复触发；2）是否真正命中缓存；3）聚合接口是否仍在内部多次重复查询。
