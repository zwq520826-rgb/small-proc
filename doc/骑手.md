## 骑手端接单功能总体计划

### 一、目标与整体流程

**目标**：实现一个完整的骑手工作闭环：
- 骑手可以在「大厅」看到待接单任务列表
- 骑手可以抢单（接单），订单从 `pending_accept` → `pending_pickup`
- 骑手可以在「我的任务」中查看自己接的单，并执行：
  - 确认取货：`pending_pickup` → `delivering`
  - 确认送达：`delivering` → `completed`

**整体数据流**（都围绕现有 `orders` 表）：
1. 用户下单：创建 `orders` 文档，`status = pending_accept`，`rider_id = null`
2. 骑手大厅：调用云对象 `order-service.getHallTasks` 拉取 `status = pending_accept` 且 `rider_id` 为空的订单
3. 骑手抢单：调用 `order-service.grabTask`，设置
   - `rider_id = 当前骑手 uid`
   - `status = pending_pickup`
   - `accept_time = Date.now()`
4. 骑手我的任务：调用 `order-service.getMyTasks` 查看 `rider_id = uid` 的订单，支持按状态筛选
5. 骑手取货：调用 `order-service.confirmPickup`，设置
   - `status = delivering`
   - `pickup_time = Date.now()`
   - （可选）`content.pickup_images`
6. 骑手送达：调用 `order-service.confirmDelivery`，设置
   - `status = completed`
   - `complete_time = Date.now()`
   - `content.delivery_images`

---

### 二、数据库设计（现有为主，后续可扩展）

#### 1. `orders` 表（已存在）

关键字段（已满足骑手侧需求）：
- `user_id`: 下单用户 uid
- `rider_id`: 骑手 uid（可为空）
- `status`: 订单状态，枚举：
  - `pending_accept`：待接单（出现在骑手大厅）
  - `pending_pickup`：待取货（骑手已接单）
  - `delivering`：配送中
  - `completed`：已完成
  - `cancelled`：已取消
- `pickup_location`: 取件地点（当前设置为“待指定”）
- `delivery_location`: 送达地址（校区 + 详细地址）
- `content.pickup_images`: 取货凭证图（可选）
- `content.delivery_images`: 送达凭证图（可选）
- `pickup_distance`: 取件距离（未来可用骑手定位计算）
- `create_time` / `accept_time` / `pickup_time` / `complete_time`

> 结论：**暂时无需新建骑手表，仅凭 `orders` 就能实现接单与任务流转**。后续如需骑手实名认证、评分等，可以再设计 `riders` 集合。

### 三、云对象接口规划（`order-service`）

> 当前代码中，这些接口大部分已经实现；这里统一梳理，以便前端调用。

#### 1. 通用：鉴权与上下文

- `_before()`: 已使用 `uni-id-common` 从 token 中解析 `uid`，并保存在 `this.uid`。
- `_checkAuth` / `checkAuth(this)`: 确保所有接口在有登录用户的前提下执行。

#### 2. 用户端（已完成）

- `createOrder(orderData)`: 用户下单
- `getOrderList(params)`: 用户查看自己的订单列表
- `getOrderDetail(orderId)`: 用户查看订单详情
- `cancelOrder(orderId)`: 用户取消订单
- `deleteOrder(orderId)`: 用户删除（已完成）订单

#### 3. 骑手端（已实现 / 待前端接入）

- `getHallTasks(params)`
  - 入参：`{ sortBy?: 'distance' | 'price', page?: number, pageSize?: number }`
  - 查询条件：
    - `status = 'pending_accept'`
    - `rider_id` 不存在或为 `null`
  - 功能：返回待接单订单列表，用于骑手大厅。

- `grabTask(orderId)`
  - 入参：`orderId`
  - 逻辑：
    1. 根据 `orderId` 查询订单
    2. 校验状态为 `pending_accept`
    3. 校验 `rider_id` 为空且 `user_id !== uid`
    4. 更新：
       - `rider_id = uid`
       - `status = 'pending_pickup'`
       - `accept_time = Date.now()`

- `getMyTasks(params)`
  - 入参：`{ status?: 'all' | 'pending_pickup' | 'delivering' | 'completed', page?: number, pageSize?: number }`
  - 查询条件：`rider_id = uid`，可按 `status` 过滤
  - 功能：骑手查看自己的任务列表。

- `confirmPickup(orderId, images = [])`
  - 入参：`orderId`, 取货凭证图片数组
  - 校验：`order.rider_id === uid` 且 `order.status === 'pending_pickup'`
  - 更新：
    - `status = 'delivering'`
    - `pickup_time = Date.now()`
    - 可选：`content.pickup_images = images`

- `confirmDelivery(orderId, images = [])`
  - 入参：`orderId`, 送达凭证图
  - 校验：`order.rider_id === uid` 且 `order.status === 'delivering'`
  - 更新：
    - `status = 'completed'`
    - `complete_time = Date.now()`
    - `content.delivery_images = images`

> 计划：短期内**不改后端，只是前端把这些接口接上**；如果在前端开发过程中发现后端字段不够再做小补充。

### 四、前端页面与交互规划

#### 1. 角色切换（已有基础）

- 位置：`pages/mine/index.vue`
  - `goBecomeRider()`: 写 `user_mode = 'rider'`，`reLaunch('/pages/rider/hall')`
  - `goClientMode()`: 写 `user_mode = 'client'`，`reLaunch('/pages/client/home')`
- 补充：
  - 在应用启动（如 `pages/client/home`、`pages/rider/hall` 的 `onLoad` / `onShow`）时，根据 `user_mode` 决定默认进入用户端或骑手端。

#### 2. 骑手大厅页面 `pages/rider/hall`

**功能**：展示待接单任务列表 + 支持按价格 / 距离排序 + 支持「一键接单」。

**逻辑规划**：
- 使用 `script setup` + `ref/computed`。
- `onLoad/onShow` 时：
  - 调用云对象：`const orderService = uniCloud.importObject('order-service')`
  - 执行 `orderService.getHallTasks({ sortBy, page, pageSize })`
  - 把返回的 `data` 映射为前端结构（如格式化时间、金额）。
- 列表项显示：
  - 送达地址（校区 + 寝室）
  - 预估金额
  - 下单时间
- `接单` 按钮：
  - 点击后调用 `orderService.grabTask(order._id)`
  - 成功后：
    - `uni.showToast('接单成功')`
    - 从大厅列表中移除该订单
    - 可选：跳转到「我的任务」详情页。

**UI 简要规划**：
- 顶部：Tab 或 Segmented 控件切换排序方式：`距离优先 / 金额优先`
- 列表：卡片式展示，每条显示关键信息 + 「接单」按钮。

#### 3. 骑手任务列表 `pages/rider/tasks/list`

**功能**：展示当前骑手的所有任务，按状态分组。

**逻辑规划**：
- 顶部 Tab：`全部 / 待取货 / 配送中 / 已完成`
- 根据选中的 `status` 调用 `getMyTasks({ status, page, pageSize })`
- 列表项简要信息：
  - 订单状态（彩色标签）
  - 送达地址 / 金额
  - 关键时间（接单时间 / 取货时间）
- 点击进入详情页：`/pages/rider/tasks/detail?id=xxx`

#### 4. 骑手任务详情页 `pages/rider/tasks/detail`

**功能**：展示单个任务详情，并提供状态操作按钮。

**逻辑规划**：
- `onLoad` 时使用 `orderService.getOrderDetail(orderId)` 拉取详情。
- 展示：
  - 收件人、电话、地址
  - 订单金额、类型
  - 当前状态与时间线
- 底部操作按钮（根据状态动态显示）：
  - `pending_pickup`：显示「确认取货」
    - 调用 `confirmPickup(orderId, images?)`
  - `delivering`：显示「确认送达」
    - 调用 `confirmDelivery(orderId, images)`
  - `completed` / `cancelled`：不展示操作按钮，仅展示信息。

> 图片上传可以沿用用户端上传组件逻辑，只是存入 `content.pickup_images` / `content.delivery_images`。

### 六、注意事项与约定

- **安全性**
  - 所有骑手端接口依赖 `_before` + `checkAuth`，必须有合法 token 才能调用。
  - 后端必须在每个接口内检查 `order.user_id` / `order.rider_id` 与 `uid` 匹配，防止越权操作。

- **状态机**
  - 严格按照：
    - 用户创建：`pending_accept`
    - 骑手接单：`pending_pickup`
    - 确认取货：`delivering`
    - 确认送达：`completed`
    - 用户取消：`cancelled`（仅限 `pending_accept`）
  - 前后端都要按这个状态机进行判断，避免非法状态跳转。

- **数据一致性**
  - 所有与骑手相关的操作都在 `orders` 单表里完成，保证骑手视角与用户视角看到的是同一条数据。
  - 前端在操作成功后要适当刷新对应列表（大厅 / 我的任务），避免显示旧状态。
